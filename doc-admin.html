Server

Starting the server

source .venv/bin/activate
python app.py --host 0.0.0.0 --port 8765

The server starts FastAPI with Uvicorn. It initialises the SQLite database, starts background tasks (transcript broadcast loop, periodic snapshot loop), and begins accepting WebSocket connections.

Set these in a .env file in the project root. The server loads it on startup.

Stopping the server

Press Ctrl+C in the terminal. Active sessions are NOT auto-ended — the session data is preserved and will be restored when clients reconnect after restart.

Warning: Restarting the server during an active session will interrupt transcription and LLM analysis. Clients will auto-reconnect but there will be a gap in the transcript.

God View

The system monitor at /admin provides a read-only view of everything happening in the system. It is designed for the system administrator who needs to observe without interfering with active sessions.

What it shows

Header: Active session ID and topic, session duration, connected client count, server uptime, WebSocket health.

Left column — Pipeline health:

  • STT: active backend, detected language, last/avg latency, chunks processed and skipped
  • LLM: active provider and model, last/avg latency, call count and errors, JSON parse success rate
  • Circuit breaker: state (closed/half-open/open), consecutive failure count
  • Token usage: cumulative input/output tokens, session cost

Center column — Live activity:

  • Knowledge graph mirror — real-time replica of the audience display
  • Node count vs 24-node budget, churn rates (additions/removals per minute)
  • Live transcript feed

Right column — System resources:

  • STT backend health (all three, with online/offline indicators, refreshed every 30s)
  • WebSocket clients and reconnect count
  • Transcript queue depth, display FPS, audio RMS, VAD state
  • Last STT output, last LLM error
  • Today's session list

Key things to watch

MetricHealthyConcern
STT Latency< 2s> 5s: STT backend struggling
LLM Latency< 20s> 40s: GPU memory pressure
Circuit Breakerclosedopen: LLM failing, auto-recovering
Parse OK / FailMostly OKHigh fail rate: LLM producing garbage
Transcript Queue0> 5: processing can't keep up
Display FPS> 30< 15: display browser struggling
Note: The god view is purely read-only. It connects as a regular WebSocket client and receives the same broadcasts as the display. It cannot affect running sessions.

GPU & Memory Management

Checking GPU status

nvidia-smi

Shows GPU utilisation, temperature, and per-process memory usage. On the DGX Spark, all services share unified memory — there's no separate GPU VRAM.

Checking Ollama

# Currently loaded models
curl -s localhost:11434/api/ps | python3 -m json.tool

# Available models
curl -s localhost:11434/api/tags | python3 -m json.tool

Memory pressure

If LLM latency is high (>40s with a full graph), GPU memory is likely saturated. Actions:

  1. Stop unused STT backends (see below)
  2. Check if multiple Ollama models are loaded (each takes 30-40 GB)
  3. Check swap usage: free -h — if swap is >5 GB used, memory is tight

System overview

# CPU load and memory
uptime && free -h

# Top memory consumers
ps aux --sort=-%mem | head -10

# Ollama process
ps -o pid,lstart,etime,rss,cmd -p $(pgrep ollama)

STT Backends

faster-whisper (port 8766)

Native Python process. 99 languages. The default backend. Always running.

Parakeet TDT (port 8010)

Docker container. 25 European languages. Highest throughput.

docker start parakeet-asr    # Start
docker stop parakeet-asr     # Stop (frees ~3 GB GPU)
docker logs parakeet-asr     # Check logs

Canary 1b v2 (port 8011)

Docker container. 25 European languages. Best accuracy but poor Norwegian auto-detection.

docker start canary-asr      # Start
docker stop canary-asr       # Stop (frees ~10 GB GPU)
docker logs canary-asr       # Check logs
Known issue: Canary's auto-language-detection fails on Norwegian and struggles even with manual language setting. For Norwegian sessions, use faster-whisper.

Health check

The server checks backend health via the /v1/stt/backends endpoint. The god view shows online/offline status for each backend, refreshed every 30 seconds.

Recommended configuration for events

Stop Canary and Parakeet before events to maximise GPU memory for Ollama:

docker stop canary-asr parakeet-asr

This frees ~13 GB of GPU memory and should bring LLM latency back under 20s.

Database

All data is in livemind.db (SQLite with WAL mode) in the project directory.

Tables

TableContents
sessionsSession ID, topic, timestamps, summary, archived flag
segmentsEvery transcript chunk with STT metadata (language, backend, latency, cleaned text)
snapshotsFull graph state at points in time (periodic, analysis, end)
actionsUser interactions (pin, rename, merge, etc.)
recapsAI-generated session recaps
synthesis_recapsCross-session synthesis recaps

Querying directly

sqlite3 livemind.db

-- List sessions
SELECT id, topic, created_at, segment_count FROM sessions ORDER BY created_at DESC;

-- Check STT language detection for a session
SELECT text, stt_language, stt_backend, stt_latency_ms
FROM segments WHERE session_id = 'XXXX' AND is_partial = 0 ORDER BY seq;

-- Check recap
SELECT recap_json FROM recaps WHERE session_id = 'XXXX';

Full reset

Stop the server, delete livemind.db, restart. A new database is created automatically. Prefer using the archive feature in the UI for non-destructive cleanup.

Backup

cp livemind.db livemind-backup-$(date +%Y%m%d).db

Recovery Procedures

Server crash during session

  1. Restart the server: python app.py --host 0.0.0.0 --port 8765
  2. All clients (display, monitor) auto-reconnect
  3. The display will receive the latest snapshot and restore the graph
  4. There will be a gap in the transcript for the downtime period
  5. The session is NOT auto-ended — it continues from where it left off

Ollama not responding

  1. Check: curl -s localhost:11434/api/ps | python3 -m json.tool
  2. Restart if needed: sudo systemctl restart ollama
  3. The model takes ~30s to reload into GPU memory on first request
  4. The circuit breaker will auto-recover once Ollama responds

STT backend down

  1. Check: docker ps | grep asr
  2. Restart: docker start canary-asr or docker start parakeet-asr
  3. Or switch to a running backend via the monitor panel

High swap usage

free -h           # Check swap
docker stop canary-asr parakeet-asr   # Free GPU memory
# If still high, consider restarting Ollama to release cached models

Complete system restart

  1. Stop Mímir: Ctrl+C
  2. Stop optional backends: docker stop canary-asr parakeet-asr
  3. Restart Ollama: sudo systemctl restart ollama
  4. Start Mímir: source .venv/bin/activate && python app.py --host 0.0.0.0 --port 8765
  5. Open /monitor and start a new session
